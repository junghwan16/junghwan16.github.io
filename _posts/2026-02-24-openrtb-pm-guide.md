> **💡 Tip:** 프로그래매틱 광고를 처음 접하는 PM이라면, **부동산 중개 플랫폼**을 떠올려 보세요.
> - **매체(Publisher)** = 임대인. "이 위치에 광고판 자리가 있어요"
> - **광고주(Advertiser)** = 임차인. "이 자리에 우리 광고를 걸고 싶어요"
> - **SSP** = 임대인 측 중개사. 최대한 비싸게 팔려고 합니다
> - **DSP** = 임차인 측 중개사. 최대한 효율적으로 사려고 합니다
> - **OpenRTB** = 중개 계약서의 표준 양식. 모든 중개사가 같은 양식을 사용합니다
>
> 이 문서는 PM이 알아야 할 **비즈니스 핵심**만 정리한 가이드입니다.

Product Manager가 OpenRTB를 이해하고, 개발팀과 효과적으로 협업하며, 비즈니스 의사결정을 내리기 위한 가이드.
코드보다 **비즈니스 맥락, 의사결정 포인트, 핵심 지표**에 초점을 맞춘다.

## 1. PM이 OpenRTB를 알아야 하는 이유

```
PM의 일상적 질문들:

"왜 Fill Rate이 떨어졌지?"
  → imp.bidfloor이 너무 높거나, 비더 타임아웃이 늘었거나, Privacy 규제 영향

"이 DSP 연동하는 데 얼마나 걸려?"
  → OpenRTB 표준 준수 여부에 따라 1주~1달 차이

"CTV 인벤토리를 지원하려면?"
  → OpenRTB 2.6의 Ad Pods + DOOH 객체 지원 필요

"매출이 왜 리포트마다 다르지?"
  → nurl(낙찰 알림) vs burl(과금 시점) vs impression(노출) 기준 차이
```

**PM이 OpenRTB를 이해하면:**
- 개발팀과 **같은 언어**로 소통할 수 있다
- 기능 우선순위를 **기술적 난이도**와 함께 판단할 수 있다
- 비즈니스 지표 이상의 **근본 원인**을 파악할 수 있다
- 파트너(DSP/SSP) 연동 범위를 **정확하게** 정의할 수 있다

## 2. 30초 만에 이해하는 OpenRTB

```
1. 유저가 앱/웹을 연다
2. 광고 지면이 있다 → SSP가 "이런 지면 있어요" (BidRequest)
3. 여러 DSP가 동시에 "얼마에 살게요" (BidResponse)
4. 가장 높은 가격이 낙찰 → 광고 노출
5. 이 모든 게 100~300ms 안에 일어남

OpenRTB = 이 과정의 표준 통신 규격
```

### 핵심 플레이어와 돈의 흐름

```
광고주 ──$──→ DSP ──$──→ SSP/Exchange ──$──→ 퍼블리셔
(광고비)      (수요)      (거래소)           (매체)

         ←── 광고 노출 ←── 광고 지면 ←── 트래픽 ──
```

| 플레이어 | 역할 | 수익 모델 | PM 관점 |
|---------|------|----------|--------|
| **광고주** | 광고 집행 | 광고비 지출 | 최종 고객 |
| **DSP** | 자동 입찰 | 광고비의 10~20% 수수료 | 수요 파트너 |
| **SSP/Exchange** | 인벤토리 판매 | 매출의 10~30% 수수료 | **우리 위치 (대부분)** |
| **퍼블리셔** | 트래픽 제공 | 광고 수익 분배 | 공급 파트너 |

## 3. PM이 알아야 할 핵심 개념

### 3.1 경매 방식

| 방식 | 설명 | PM 포인트 |
|------|------|----------|
| **1st Price** | 최고 입찰가 = 지불 금액 | 현재 업계 표준. DSP가 bid shading 전략 사용 |
| **2nd Price** | 최고 입찰자가 2등 가격+α 지불 | 과거 표준. SSP 조작 가능성으로 퇴조 |

> **PM 액션:** 자사 시스템의 경매 방식 확인. 대부분 1st Price. `at` 필드로 구분 (1=1st, 2=2nd)

### 3.2 가격 체계

```
모든 가격은 CPM (Cost Per Mille) = 1000회 노출당 비용

예시:
  price: 2.50 → 1000회 노출에 $2.50
  1회 노출 = $0.0025

Floor Price (bidfloor):
  "이 가격 이하로는 안 팔아요"
  → Floor이 높으면: 매출↑ 하지만 Fill Rate↓
  → Floor이 낮으면: Fill Rate↑ 하지만 매출↓
```

> **PM 액션:** Floor Price 전략은 매출에 직결. 인벤토리 유형별 최적 Floor을 A/B 테스트해야 한다.

### 3.3 미디어 타입

| 타입 | 설명 | CPM 범위 | 복잡도 | PM 우선순위 |
|------|------|---------|--------|-----------|
| **Banner** | 고정 크기 이미지/HTML | $0.5~5 | 낮음 | 기본 지원 |
| **Video** | VAST 기반 영상 광고 | $5~30+ | 높음 | 매출 임팩트 큼 |
| **Native** | 콘텐츠 통합형 | $1~10 | 중간 | UX 품질 높음 |
| **Audio** | 오디오 스트리밍 광고 | $5~15 | 중간 | 니치 시장 |

> **PM 액션:** Video가 CPM이 가장 높지만 구현 복잡도도 높다. Banner → Native → Video 순으로 지원 확장을 권장.

### 3.4 인벤토리 유형

| 유형 | OpenRTB 객체 | 특징 | 시장 트렌드 |
|------|-------------|------|-----------|
| **Web** | `site` | 전통적 웹사이트 | 안정적, 쿠키 제한 이슈 |
| **Mobile App** | `app` | 앱 내 광고 | 성장세, IDFA/GAID 제한 |
| **CTV** | `app` + video pods | 커넥티드 TV | 고성장, 고CPM |
| **DOOH** | `dooh` (2.6 신규) | 디지털 옥외광고 | 신규 시장, 표준화 진행 중 |

### 3.5 Private Marketplace (PMP)

```
Open Auction:                    PMP Deal:
┌────────────────────┐          ┌────────────────────┐
│ 모든 DSP 참여      │          │ 초대된 DSP만 참여   │
│ 공개 경쟁           │          │ 사전 협의 가격       │
│ bidfloor로 최소가   │          │ deal.bidfloor       │
│                    │          │ 보장형/선호형        │
│ Fill Rate: 높음     │          │ Fill Rate: 낮지만   │
│ CPM: 변동           │          │ CPM: 안정/높음      │
└────────────────────┘          └────────────────────┘
```

| PMP 유형 | 설명 | PM 포인트 |
|---------|------|----------|
| **Preferred Deal** | 고정가, 구매 의무 없음 | DSP가 패스하면 Open으로 넘어감 |
| **Private Auction** | 초대된 DSP끼리 경쟁 | 프리미엄 인벤토리에 적합 |
| **Programmatic Guaranteed** | 고정가+고정량 보장 | 전통 직거래의 자동화 |

> **PM 액션:** 프리미엄 인벤토리에 PMP Deal을 적용하면 CPM 2~5배 상승 가능. 영업팀과 협업 필요.

## 4. 핵심 비즈니스 지표 (KPI)

### 4.1 매출 관련

| 지표 | 산식 | 의미 | 건강한 범위 |
|------|------|------|-----------|
| **Revenue** | SUM(win_price) / 1000 | 총 매출 | - |
| **eCPM** | Revenue / Impressions × 1000 | 유효 CPM | $1~15 (유형별) |
| **ARPDAU** | Revenue / DAU | 유저당 일 매출 | $0.01~0.10 |
| **Fill Rate** | Filled Impressions / Total Requests | 광고 채움 비율 | 60~90% |
| **Win Rate** | Wins / Bids | 낙찰 비율 | 10~30% (DSP) |

### 4.2 기술 관련 (개발팀에게 요청할 지표)

| 지표 | 의미 | PM이 알아야 할 이유 |
|------|------|-------------------|
| **QPS** | 초당 처리 요청 수 | 트래픽 대응 용량 |
| **Latency p99** | 99분위 응답 시간 | tmax 초과 시 입찰 불참 = 매출 손실 |
| **Timeout Rate** | 비더 타임아웃 비율 | 5% 초과 시 DSP 연동 점검 필요 |
| **Bid Rate** | 입찰 비율 | 급감 시 DSP 이슈 또는 인벤토리 품질 문제 |
| **Error Rate** | HTTP 에러 비율 | 1% 초과 시 기술적 문제 |
| **Discrepancy** | SSP-DSP 간 수치 차이 | 5% 이내가 정상 |

### 4.3 KPI 이상 징후와 원인 진단

```
Fill Rate 하락
├── bidfloor이 너무 높음 → Floor 조정
├── DSP 타임아웃 증가 → 서버 성능/네트워크 점검
├── Privacy 규제 변경 → GDPR/CCPA 영향 분석
├── 인벤토리 품질 하락 → IVT(부정 트래픽) 점검
└── 시즈널리티 → 광고 시장 계절 변동 (Q1 하락, Q4 상승)

eCPM 하락
├── 경쟁 DSP 감소 → 비더 수 확인, 신규 연동 추진
├── 오디언스 품질 하락 → User ID 매칭률 확인
├── 인벤토리 믹스 변화 → Banner vs Video 비중 확인
├── 시장 전반 하락 → 업계 벤치마크 비교
└── Floor 전략 이슈 → Floor 최적화 A/B 테스트
```

## 5. DSP/SSP 연동 관리

### 5.1 연동 체크리스트 (PM용)

**연동 전 확인:**

- [ ] 파트너가 OpenRTB 어떤 버전을 지원하는가? (2.5 vs 2.6)
- [ ] 지원하는 미디어 타입은? (Banner/Video/Native/Audio)
- [ ] 지원하는 인벤토리 타입은? (Web/App/CTV)
- [ ] 지역 제한이 있는가? (한국만/글로벌)
- [ ] `ext` 필드에 커스텀 요구사항이 있는가?
- [ ] 테스트 환경(Sandbox)을 제공하는가?
- [ ] SLA(응답 시간, 가용성)를 보장하는가?

**연동 중 확인:**

- [ ] 개발팀이 BidRequest/BidResponse 매핑을 완료했는가?
- [ ] 테스트 입찰이 정상적으로 오가는가?
- [ ] 매크로 치환 (`${AUCTION_PRICE}`)이 올바른가?
- [ ] Privacy 필드 전달이 정상인가?

**연동 후 모니터링:**

- [ ] Bid Rate가 기대 수준인가? (최소 10%+)
- [ ] Timeout Rate가 5% 이하인가?
- [ ] Discrepancy(수치 차이)가 5% 이내인가?
- [ ] 크리에이티브 품질 이슈가 없는가?
- [ ] Revenue가 기대 수준인가?

### 5.2 연동 일정 가이드

| 단계 | 소요 기간 | 설명 |
|------|----------|------|
| 기술 검토 | 1~2일 | 스펙 호환성, ext 요구사항 파악 |
| 개발 | 3~10일 | 어댑터 구현 (표준 준수 시 빠름) |
| 테스트 | 2~5일 | 샌드박스 테스트, 에러 수정 |
| 소프트 론칭 | 1~3일 | 트래픽 1~5%로 시작 |
| 전체 론칭 | 1일 | 트래픽 100% 전환 |
| 최적화 | 지속 | Floor, 타임아웃, 필터링 튜닝 |

> **PM 액션:** 표준 OpenRTB를 준수하는 파트너는 1~2주, 커스텀 요구사항이 많은 파트너는 3~4주 예상.

### 5.3 파트너 평가 기준

| 기준 | 가중치 | 측정 방법 |
|------|--------|----------|
| **Bid Rate** | 높음 | 입찰 비율 (높을수록 좋음) |
| **eCPM** | 높음 | 평균 CPM (높을수록 좋음) |
| **Fill Quality** | 중간 | 크리에이티브 품질, 브랜드 세이프티 |
| **Timeout Rate** | 중간 | 타임아웃 비율 (낮을수록 좋음) |
| **Discrepancy** | 중간 | SSP-DSP 수치 차이 (5% 이내) |
| **기술 지원** | 낮음 | 이슈 대응 속도 |

## 6. 의사결정 가이드

### 6.1 OpenRTB 버전 선택

```
OpenRTB 2.5 vs 2.6:

2.5: 안정적, 대부분의 DSP 지원, 충분한 기능
2.6: CTV Ad Pods, DOOH, Structured UA 추가

판단 기준:
├── CTV 인벤토리가 있는가?        → Yes → 2.6 필요
├── DOOH 인벤토리가 있는가?       → Yes → 2.6 필요
├── Chrome UA 축소 대응이 급한가?  → Yes → 2.6의 sua 객체 필요
└── 위 모두 No                   → 2.5로 충분, 점진적 마이그레이션
```

### 6.2 기능 우선순위 매트릭스

| 기능 | 매출 임팩트 | 개발 난이도 | 우선순위 |
|------|-----------|-----------|---------|
| Banner 지원 | 중 | 하 | **P0 - 즉시** |
| Video (VAST) 지원 | 상 | 상 | **P0 - 즉시** |
| Native 지원 | 중 | 중 | **P1 - 다음** |
| PMP Deal | 상 | 중 | **P1 - 다음** |
| Floor 최적화 | 상 | 중 | **P1 - 다음** |
| Privacy 준수 | 필수 | 중 | **P0 - 필수** |
| Supply Chain | 중 | 하 | **P2 - 계획** |
| Ad Pods (CTV) | 상 (CTV) | 상 | **P2 - CTV 시** |
| DOOH 지원 | 낮음 | 상 | **P3 - 나중에** |
| Audio 지원 | 낮음 | 중 | **P3 - 나중에** |

### 6.3 Revenue 최적화 레버

```
Revenue = Requests × Fill Rate × eCPM / 1000

┌──────────────────────────────────────────────────────┐
│                Revenue 높이는 방법                     │
├──────────────────────────────────────────────────────┤
│                                                      │
│  Requests ↑                                          │
│  ├── 트래픽 증가 (MAU, 페이지뷰)                       │
│  ├── 광고 슬롯 추가 (UX 균형 필요)                     │
│  └── 새로운 인벤토리 유형 (CTV, Audio)                  │
│                                                      │
│  Fill Rate ↑                                         │
│  ├── DSP 연동 수 확대 (경쟁 ↑)                         │
│  ├── Floor Price 최적화 (너무 높으면 역효과)             │
│  ├── 타임아웃 튜닝 (tmax 적절히)                        │
│  └── 인벤토리 품질 개선 (IVT 제거)                      │
│                                                      │
│  eCPM ↑                                              │
│  ├── Video 인벤토리 확대 (CPM 5~10x)                   │
│  ├── PMP Deal 활성화 (프리미엄 가격)                    │
│  ├── User ID 매칭률 향상 (타겟팅 ↑ → 가격 ↑)           │
│  ├── 데이터 세그먼트 제공 (user.data)                   │
│  └── 헤더비딩 도입 (경쟁 극대화)                        │
│                                                      │
└──────────────────────────────────────────────────────┘
```

## 7. Privacy & 규제 (PM 관점)

### 7.1 규제별 비즈니스 영향

| 규제 | 대상 | 비즈니스 영향 | PM 액션 |
|------|------|-------------|--------|
| **GDPR** | EU 사용자 | 동의 없으면 개인화 불가 → eCPM 30~50% 하락 | CMP(동의 관리 플랫폼) 도입 |
| **CCPA** | 캘리포니아 사용자 | 옵트아웃 시 데이터 판매 금지 | `us_privacy` 필드 전달 |
| **COPPA** | 아동 대상 서비스 | 행동 타겟팅 전면 금지 | 아동용 앱이면 `coppa=1` |
| **GPP** | 글로벌 | 통합 프라이버시 표준 (신규) | 장기적 준비 |
| **IDFA/ATT** | iOS 앱 | 옵트인율 15~25% → 매칭률 급감 | 대체 ID 솔루션 검토 |
| **쿠키 제한** | Chrome 웹 | 3P 쿠키 없이 타겟팅 어려움 | Topics API, UID2 등 대안 |

### 7.2 Privacy가 매출에 미치는 영향

```
Privacy 강화 전:
  User ID 매칭률: 80% → 정확한 타겟팅 → eCPM $5.0

Privacy 강화 후:
  User ID 매칭률: 30% → 타겟팅 약화 → eCPM $2.0~3.0

대응 전략:
  ├── 1P 데이터 활용 (로그인 유저 → buyeruid)
  ├── 컨텍스트 타겟팅 (콘텐츠 기반, site.cat)
  ├── Extended IDs (user.eids → UID2, ID5 등)
  └── 코호트 기반 (Topics API 등)
```

> **PM 액션:** Privacy 규제는 피할 수 없다. Extended ID 솔루션(UID2, ID5)과 1P 데이터 전략을 로드맵에 포함해야 한다.

## 8. 개발팀과 소통할 때 쓰는 용어집

### 8.1 일상 대화에서 자주 나오는 용어

| 용어 | 발음 | 의미 | 맥락 |
|------|------|------|------|
| BidRequest | 비드리퀘스트 | 입찰 요청 | "비드리퀘스트에 video 객체 추가해야 해요" |
| BidResponse | 비드리스폰스 | 입찰 응답 | "리스폰스에 mtype이 빠져있어요" |
| Imp (Impression) | 임프/임프레션 | 광고 슬롯 | "이 페이지에 임프 3개를 보내고 있어요" |
| adm | 에이디엠 | 광고 마크업 | "adm에 VAST가 안 들어와요" |
| nurl | 엔유알엘 | Win Notice URL | "nurl 호출이 안 되고 있어요" |
| burl | 비유알엘 | Billing Notice URL | "burl 기준으로 과금해야 해요" |
| bidfloor | 비드플로어 | 최소 입찰가 | "플로어를 $2로 올려볼까요?" |
| tmax | 티맥스 | 최대 응답 시간 | "티맥스 200ms인데 타임아웃이 많아요" |
| schain | 에스체인 | 공급망 체인 | "schain 검증을 추가해야 합니다" |
| VAST | 바스트 | 비디오 광고 규격 | "VAST 파싱 에러가 나요" |
| ext | 익스트/확장 | 확장 필드 | "이 DSP는 ext에 커스텀 필드가 필요해요" |
| Deal | 딜 | PMP 거래 | "이 퍼블리셔랑 딜을 만들어야 해요" |
| No-bid | 노비드 | 입찰 미참여 | "이 DSP 노비드율이 90%예요" |
| Fill Rate | 필레이트 | 광고 채움률 | "필레이트가 어제 대비 10% 떨어졌어요" |
| Discrepancy | 디스크레펀시 | 수치 불일치 | "이 DSP랑 디스크레펀시가 15%예요" |

### 8.2 개발팀에게 질문할 때 유용한 표현

```
Fill Rate 관련:
  "이 DSP의 Bid Rate이 어떻게 되나요?"
  "Timeout Rate이 정상 범위인가요?"
  "bidfloor 변경이 Fill Rate에 얼마나 영향 주나요?"

매출 관련:
  "nurl 기준 매출과 burl 기준 매출 차이가 얼마나 되나요?"
  "이 DSP의 eCPM이 다른 DSP 대비 어떤가요?"
  "Video 인벤토리 비중을 늘리면 전체 eCPM에 얼마나 영향이 있을까요?"

연동 관련:
  "이 DSP가 OpenRTB 2.6을 지원하나요 2.5인가요?"
  "ext에 필요한 커스텀 필드가 뭔가요?"
  "연동 테스트를 위한 샌드박스 환경이 있나요?"

트러블슈팅:
  "이 DSP에서 nbr(No-bid reason) 코드를 보내주나요? 어떤 코드가 많나요?"
  "adm 검증 실패가 어떤 유형이 많나요?"
  "mtype 불일치 에러가 얼마나 발생하나요?"
```
